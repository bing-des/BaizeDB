## PG "No database selected" 修复
- **根因**：`execute_query` 前端没传 `database` 参数，PG 的查询走主连接池（连 `postgres` 库），查用户表报错
- **修复链路**：
  1. 后端 `execute_query` 已有 `database: Option<String>` 参数（之前已加），PG 分支已正确从 `db_pools` 取连接池
  2. 后端 `execute_query_paged` 增加 `database: Option<String>` 字段，传给 `execute_query`
  3. 前端 `queryApi.execute/executePaged` 增加 `database?` 参数
  4. `QueryEditor` 调用 `queryApi.execute(tab.connectionId, trimmed, tab.database)` 传 `tab.database`
  5. `openQuery` 对 PG 连接行"新建查询"时 fallback 用 `conn.database`（连接配置的默认库）
  6. `ensure_pg_db_pool` 从 `async fn` 改为 `pub async fn`（query.rs 需要跨模块调用）

## PG timestamp 类型数据不显示修复
- **根因**：`pg_to_json` 对 timestamp 类型没有专门分支，走 `try_get::<String>` 失败（sqlx 不会自动将 chrono 类型转 String）
- **更严重**：database.rs 版本用 `contains("INT")` 匹配整数类型，`TIMESTAMP` 包含 `INT` 子串被误匹配，`try_get::<i64>` 也失败 → 返回 Null
- **修复**：两个文件的 `pg_to_json` 都增加 timestamp/timestamptz/date/time/uuid/json/jsonb 分支
  - TIMESTAMP → `chrono::NaiveDateTime` → `.to_string()`
  - TIMESTAMPTZ → `chrono::DateTime<Utc>` → `.to_string()`
  - DATE → `chrono::NaiveDate` → `.to_string()`
  - TIME → `chrono::NaiveTime` → `.to_string()`
  - UUID → `uuid::Uuid` → `.to_string()`
  - JSON/JSONB → `serde_json::Value` → `.to_string()`
- database.rs 的整数匹配从 `contains("INT")` 改为精确匹配 `INT2|INT4|INT8|SERIAL|BIGSERIAL|SMALLSERIAL`

## MySQL 预处理语句协议错误修复
- **错误**：执行 `SELECT * FROM tb_person;` 报 “This command is not supported in the prepared statement protocol yet”
- **根因**：MySQL 的 `USE` 语句不支持预处理语句协议，sqlx 默认对 `sqlx::query` 使用预处理语句
- **修复**：
  1. 在 `execute_query` 的 MySQL 分支中，若指定了 `database`，则从连接池获取一个独立连接
  2. 使用 `conn.execute(&*use_sql)`（`Executor` trait 的原始执行方法）执行 `USE` 语句，绕过预处理协议
  3. 使用同一个连接执行后续查询（`SELECT` 等），确保数据库上下文正确
  4. 未指定 `database` 时仍使用连接池，避免额外开销
- **修改文件**：`src-tauri/src/commands/query.rs`
  - 添加 `use sqlx::Executor;`
  - 重构 MySQL 分支逻辑，区分有无 database 的情况
- **编译**：Rust 和 TypeScript 编译均通过
