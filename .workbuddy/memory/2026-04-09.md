## PG "No database selected" 修复
- **根因**：`execute_query` 前端没传 `database` 参数，PG 的查询走主连接池（连 `postgres` 库），查用户表报错
- **修复链路**：
  1. 后端 `execute_query` 已有 `database: Option<String>` 参数（之前已加），PG 分支已正确从 `db_pools` 取连接池
  2. 后端 `execute_query_paged` 增加 `database: Option<String>` 字段，传给 `execute_query`
  3. 前端 `queryApi.execute/executePaged` 增加 `database?` 参数
  4. `QueryEditor` 调用 `queryApi.execute(tab.connectionId, trimmed, tab.database)` 传 `tab.database`
  5. `openQuery` 对 PG 连接行"新建查询"时 fallback 用 `conn.database`（连接配置的默认库）
  6. `ensure_pg_db_pool` 从 `async fn` 改为 `pub async fn`（query.rs 需要跨模块调用）

## PG timestamp 类型数据不显示修复
- **根因**：`pg_to_json` 对 timestamp 类型没有专门分支，走 `try_get::<String>` 失败（sqlx 不会自动将 chrono 类型转 String）
- **更严重**：database.rs 版本用 `contains("INT")` 匹配整数类型，`TIMESTAMP` 包含 `INT` 子串被误匹配，`try_get::<i64>` 也失败 → 返回 Null
- **修复**：两个文件的 `pg_to_json` 都增加 timestamp/timestamptz/date/time/uuid/json/jsonb 分支
  - TIMESTAMP → `chrono::NaiveDateTime` → `.to_string()`
  - TIMESTAMPTZ → `chrono::DateTime<Utc>` → `.to_string()`
  - DATE → `chrono::NaiveDate` → `.to_string()`
  - TIME → `chrono::NaiveTime` → `.to_string()`
  - UUID → `uuid::Uuid` → `.to_string()`
  - JSON/JSONB → `serde_json::Value` → `.to_string()`
- database.rs 的整数匹配从 `contains("INT")` 改为精确匹配 `INT2|INT4|INT8|SERIAL|BIGSERIAL|SMALLSERIAL`

## MySQL 预处理语句协议错误修复
- **错误**：执行 `SELECT * FROM tb_person;` 报 “This command is not supported in the prepared statement protocol yet”
- **根因**：MySQL 的 `USE` 语句不支持预处理语句协议，sqlx 默认对 `sqlx::query` 使用预处理语句
- **修复**：
  1. 在 `execute_query` 的 MySQL 分支中，若指定了 `database`，则从连接池获取一个独立连接
  2. 使用 `conn.execute(&*use_sql)`（`Executor` trait 的原始执行方法）执行 `USE` 语句，绕过预处理协议
  3. 使用同一个连接执行后续查询（`SELECT` 等），确保数据库上下文正确
  4. 未指定 `database` 时仍使用连接池，避免额外开销
- **修改文件**：`src-tauri/src/commands/query.rs`
  - 添加 `use sqlx::Executor;`
  - 重构 MySQL 分支逻辑，区分有无 database 的情况
- **编译**：Rust 和 TypeScript 编译均通过

  3. **编译验证**：`cargo check` 零错误，仅有无用代码警告
- **影响**：解决了 PostgreSQL 作为目标数据库时，时间戳类型列的数据迁移问题，确保 chrono 类型正确绑定为 PostgreSQL 原生时间戳类型，而非文本

## PostgreSQL 时间戳类型绑定修复

- **问题**：迁移报错 "error returned from database: column 'create_time' is of type timestamp without time zone but expression is of type text"
- **根因**：PostgreSQL 是强类型数据库，插入时需要根据表字段类型对数据类型进行处理。在 `postgres_target.rs` 的 `insert_rows` 方法中，绑定参数时没有根据目标列的数据类型进行适配，导致时间戳值可能被绑定为文本类型。
- **修复**：
  1. **同时迭代列和值**：修改 `insert_rows` 方法，同时遍历 `schema.columns` 和 `row.values`，确保每个值绑定到对应的列类型。
  2. **时间戳类型适配**：
     - 对于 `Value::Timestamp`（带时区），根据目标列类型决定绑定类型：
       - 目标列是 `timestamp with time zone`：直接绑定 `chrono::DateTime<Utc>`
       - 目标列是 `timestamp without time zone`：转换为 `chrono::NaiveDateTime`（使用 `naive_utc()`）
     - 对于 `Value::DateTime`（不带时区）：
       - 目标列是 `timestamp with time zone`：转换为 `chrono::DateTime<Utc>`（使用 `TimeZone::from_utc_datetime()`）
       - 目标列是 `timestamp without time zone`：直接绑定 `chrono::NaiveDateTime`
  3. **导入 TimeZone trait**：添加 `use chrono::TimeZone;` 以使用 `from_utc_datetime` 方法。
- **修改文件**：`src-tauri/src/commands/migration/postgres_target.rs`
- **编译状态**：`cargo check` 零错误，仅剩配置文件警告。
- **预期效果**：解决 PostgreSQL 时间戳类型不匹配问题，确保迁移过程中时间戳值正确绑定为原生 PostgreSQL 时间戳类型，而非文本。

## PostgreSQL 迁移 timestamp 类型不匹配修复（第二轮）

- **问题**：仍然报 "column 'create_time' is of type timestamp without time zone but expression is of type text"
- **根因分析**：
  1. **NULL 值绑定类型错误**：`Value::Null` 统一绑定为 `Option<String>::None`，PostgreSQL 预处理语句会推断参数类型为 `text`，与 `timestamp` 列不兼容
  2. **字符串值未正确转换**：MySQL 数据源提取 datetime 值时，如果 `try_get::<NaiveDateTime>` 失败，值变成 `Value::Null` 或 fallback 到 `Value::String`，但绑定到目标 timestamp 列时没有正确转换
  3. **datetime 解析格式不全**：只支持 `%Y-%m-%d %H:%M:%S`，不支持带毫秒格式
- **修复**：
  1. **NULL 值类型感知绑定**：`Value::Null` 根据 `bind_data_type` 绑定对应类型的 `Option<T>::None`，例如 timestamp 列绑定 `Option<NaiveDateTime>::None`
  2. **MySQL/PG 数据源字符串 fallback**：`read_table_data` 中 datetime 类型的 `try_get` 链末尾添加字符串 fallback，确保值不丢失
  3. **增强字符串→时间戳解析**：新增 `parse_naive_datetime` 方法，支持多种格式（含毫秒、ISO 8601、斜杠分隔等）
  4. **`Value::String` 完整类型转换**：根据目标列类型将字符串转为布尔、整数、浮点数、日期、时间、时间戳等
  5. **`Value::Decimal` 完整类型转换**：根据目标列类型将十进制字符串转为整数、浮点数等
- **修改文件**：
  - `src-tauri/src/commands/migration/postgres_target.rs`：NULL 值类型感知绑定、字符串值完整类型转换、parse_naive_datetime 方法
  - `src-tauri/src/commands/migration/mysql_source.rs`：datetime 类型字符串 fallback

## PG 迁移数据丢失根因修复（连接池 search_path 丢失）

- **问题**：迁移执行完毕但数据库中没有数据
- **根因**：sqlx `PgPool` 是连接池，`SET search_path` 和 `INSERT` 可能分配到不同连接，导致 INSERT 时 search_path 已丢失，数据插到了错误的 schema 或静默失败
- **修复**：所有 PG 操作（create_table、add_column_comment、insert_rows、truncate_table）改用事务（`pool.begin()`），确保 `SET search_path` 和后续 SQL 在同一个连接上执行，最后 `tx.commit()`
- **附加修复**：`inserted` 计数从无条件 +1 改为 `rows_affected()`，PG 和 MySQL 均已修复

- **问题**：`incorrect binary data format in bind parameter` — MySQL 的 tinyint 值（如 63）被标为 `TinyInt`，但 PG 列是 `BigInt`，绑定时 `TinyInt(i8)` 固定绑成 `Int2`，PG 期望 `Int8`
- **根因**：`Value::TinyInt/SmallInt/Integer` 绑定时不看 `bind_data_type`，直接按自身 Rust 类型绑
- **修复**：所有整数 Value 根据 `bind_data_type` 提升到正确的 PG 类型（TinyInt→i16/i32/i64, SmallInt→i32/i64, Integer→i64），Float→Double 也处理

## PG 迁移数据写入错误数据库修复（search_path 不能切换 database）

- **问题**：迁移数据写入到了 `admin` 库而不是用户指定的目标库
- **根因**：PostgreSQL 的 `SET search_path` 只能切换 **schema**，不能切换 **database**！之前的实现错误地认为 `SET search_path TO target_db, public` 能切换数据库，实际上它只是把 `target_db` 当 schema 名去搜索。连接池创建时连的 `admin` 库，所有数据都写入 `admin` 库的 `public` schema
- **修复方案**：为 PostgreSQL 目标/源数据库创建独立的连接池（复用已有的 `ensure_pg_db_pool` 机制），彻底移除 `SET search_path`
- **改动文件**：
  1. `migration.rs`：`create_data_source` 和 `create_data_target` 增加 `database` 参数，PG 类型调用 `ensure_pg_db_pool` 创建指向目标数据库的独立连接池
  2. `postgres_target.rs`：移除所有 `SET search_path` 和事务包装，SQL 改用 `public.table` 全限定名（连接池已连到目标数据库）
  3. `postgres_source.rs`：移除 `SET search_path`，SQL 改用 `public.table` 全限定名
